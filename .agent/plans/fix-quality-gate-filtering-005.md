## Objective

Fix two residual bugs in the PRD skill's quality gate filtering that cause false-positive gates on doc-only stories:
1. **Fragile keyword detection** — negation phrases and absent doc-only keywords bypass filtering
2. **Tag stripping in metadata** — `[code-only]`/`[ui-only]` tags dropped from Document Metadata Quality Gates list

## Step 1 — Fix classification rule in SKILL.md (US-005-01)

### 1a. Add negation-awareness to the `[code-only]` filter

**Target:** SKILL.md lines 247-250 (the `[code-only]` gate filtering rule)

**Current (line 247):**
```markdown
- Gates tagged `[code-only]` → skipped when format rationale contains any of: `documentation`, `spec`, `wiki`, `markdown` AND contains none of: "UI", "interface", "browser", "code", "implementation"
```

**After:**
```markdown
- Gates tagged `[code-only]` → skipped when format rationale contains any of: `documentation`, `spec`, `wiki`, `markdown` AND contains no **affirming** code/UI signals. Negation phrases ("no executable code", "no code produced", "not code-producing", "no implementation") do NOT count as code signals — only affirmative references to code production trigger the mixed-signal rule.
```

**Satisfies:** AC-005-01 (negation phrases don't trigger mixed-signal), AC-005-03 (doc-only with negated code = doc-only)

### 1b. Add rationale rewrite obligation for absent doc-only keywords

**Target:** SKILL.md line 248 (absent rationale rule) and line 250 (story-type classification)

**Current (line 250):**
```markdown
**Story-type classification rule:** A story is doc-only only when its format rationale contains doc-only keywords (`documentation`, `spec`, `wiki`, `markdown`) AND contains no code-producing or UI-deliverable signals [...]
```

**After (R1-F2 resolved — concrete inference criteria):**
```markdown
**Story-type classification rule:** A story is doc-only when EITHER:
1. Its format rationale contains doc-only keywords (`documentation`, `spec`, `wiki`, `markdown`) AND contains no affirming code/UI signals, OR
2. Its format rationale lacks doc-only keywords — the generator MUST then classify the story by checking: does the story produce an executable artifact (code, script, config that runs)? If NO, the generator MUST rewrite the rationale to include at least one doc-only keyword (`documentation`, `spec`, `wiki`, or `markdown`) before applying the filter.

**Rationale rewrite obligation:** When the generator produces a format rationale that describes a non-executable deliverable but omits doc-only keywords, the generator must insert a doc-only keyword into the rationale. The keyword must accurately describe the deliverable type (e.g., "specification" for spec changes, "documentation" for docs, "markdown" for template edits). This closes the gap where the generator and filter are out of sync.

**Classification test:** Does this story's deliverable get compiled, interpreted, linted, typechecked, or tested by a CI pipeline? If yes → code story. If no → doc-only.

Mixed-signal rationales (doc-only keyword + affirming code/UI signal, e.g., "spec change that also requires code migration") are treated as code stories: `[code-only]` gates are included.
```

**Satisfies:** AC-005-02 (absent keywords → concrete inference + rewrite), AC-005-04 (mixed signals preserved)

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| "Rationale rewrite obligation" adds complexity to the generator | The rewrite is a single-step check: if no doc-only keyword present AND story is non-executable, insert one keyword. Not semantic analysis. |
| Negation detection may miss novel phrasing ("without code", "excludes implementation") | Two-layer defense: (1) negation-awareness handles common negation patterns, (2) the rewrite obligation ensures a doc-only keyword is always present before the filter runs. Residual risk: a novel negation containing an affirming code signal substring (e.g., "no code but also implementation notes") could still trigger mixed-signal. This is accepted as low-probability; the rewrite obligation is the primary safeguard. |
| Tagging non-default preset gates may change behavior for users on python/rust/go profiles | Only code-tooling gates are tagged; the filter only activates for doc-only stories. Code-producing stories are unaffected. This is the intended behavior — doc-only stories shouldn't have `Ruff passes` gates either. |
| Template changes cause diff noise on regenerated PRDs | Only the metadata section changes — story content and ACs unchanged |
