# Plan: JX QA Onboarding Documentation

## Summary

Create a durable wiki onboarding page for QA users at `wiki/code/JX QA Onboarding.md`. The page should guide a QA through installing `jx-qa` and `jx-kb` from the GitHub marketplace, preparing a local QA project folder, initializing the local wiki/knowledge base, installing dependencies, using the plugin workflow, using Git safely, and reporting issues.

Use `jairosoft-com/jodex-plugins` as the canonical marketplace repo. Use `jx-qa@jodex-plugins` for QA automation and `jx-kb@jodex-plugins` for the local onboarding/project knowledge base. Treat `jodex-qa-ai` as legacy/repo-alias wording only when explaining stale references.

## Documentation Content

The onboarding page must include:

- Audience and outcome: a QA user can set up a local machine, install `jx-qa` and `jx-kb`, initialize a project wiki, generate test plans/specs, run Playwright tests, and report problems.
- Prerequisites:
  - Claude Code CLI or Desktop with plugin support.
  - Git.
  - Python 3 plus `openpyxl`.
  - Node.js 18+.
  - Playwright browsers.
  - `playwright-cli`.
- Recommended local QA project layout:
  ```text
  <qa-project>/
  ├── raw/articles/       # BRD/PRD markdown inputs
  ├── test-plans/         # .xlsx test plans
  ├── tests/              # generated Playwright .spec.ts files
  ├── wiki/               # generated by /jx-kb:init wiki
  ├── test-results/       # generated, do not commit
  └── playwright-report/  # generated, do not commit
  ```
  - Tell the QA to create only the project-owned working folders they need, such as `raw/articles/`, `test-plans/`, and `tests/`.
  - Do not instruct the QA to manually create `wiki/_schema.md`, taxonomy folders, `_index.md`, `_log.md`, `_backlog.md`, or `wiki/raw/sources/`; those are generated by `/jx-kb:init wiki`.
- Git setup and workflow:
  - Clone or initialize the QA/project repo.
  - Create a branch per onboarding, test extraction, or spec-generation task.
  - Create or update `.gitignore` before installing dependencies or running tests.
  - Commit source docs, test plans, and generated specs intentionally.
  - Do not commit `node_modules/`, `.venv/`, `test-results/`, `playwright-report/`, `blob-report/`, or local `.env` files.
  - Include this minimum `.gitignore` template in the onboarding doc:
    ```gitignore
    node_modules/
    .venv/
    venv/
    test-results/
    playwright-report/
    blob-report/
    playwright/.cache/
    .env
    .env.local
    ```
- Marketplace install:
  ```text
  /plugin marketplace add jairosoft-com/jodex-plugins
  /plugin install jx-qa@jodex-plugins
  /plugin install jx-kb@jodex-plugins
  /reload-plugins
  ```
- Dependency install:
  ```bash
  python3 -m pip install openpyxl
  npm install -D @playwright/test
  npx playwright install
  npm install -g @playwright/cli@latest
  ```
- Knowledge base initialization:
  ```text
  /jx-kb:init wiki
  ```
  - Make this the only setup step for wiki scaffolding.
  - Explain that `/jx-kb:init wiki` automatically creates `wiki/_schema.md`, `wiki/_index.md`, `wiki/_log.md`, `wiki/_backlog.md`, taxonomy directories, and `wiki/raw/sources/`.
  - If `wiki/_schema.md` already exists, do not blindly reinitialize; the skill asks before resetting schema files and preserving existing pages.
  - After initialization, QA can use `/jx-kb:ingest <source> wiki` to file BRDs, setup notes, or troubleshooting writeups and `/jx-kb:query <question> wiki` to retrieve them.
- First workflow:
  ```text
  /jx-kb:init wiki
  /jx-qa:extract raw/articles/<BRD_OR_PRD>.md
  /jx-qa:generate test-plans/<test-plan>.xlsx
  /jx-qa:test
  /jx-qa:test ui
  /jx-qa:test headed
  ```
- Issue reporting:
  - Report plugin/setup issues through GitHub Issues on `jairosoft-com/jodex-plugins`.
  - Include OS, shell, Claude Code version, plugin install command, Node/Python versions, exact command run, expected behavior, actual behavior, error output, and sanitized sample files when useful.
  - Separate product defects from plugin/tooling issues: product behavior belongs in the product tracker; plugin install/generation/runtime problems belong in the plugin repo.

## Assumptions

- The target reader is a QA user installing and using `jx-qa` plus `jx-kb`, not a plugin developer.
- The primary install path is GitHub marketplace install, not local checkout.
- The doc is cross-platform, using macOS/Linux shell commands as baseline and adding Windows/WSL notes only where necessary.
- GitHub Issues on `jairosoft-com/jodex-plugins` is the canonical issue-reporting route for plugin and setup problems.
- The onboarding guide belongs in the maintained wiki first, not the root README.
