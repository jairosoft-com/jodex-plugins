## Part I: Strategic Foundation (BRD)

### 1. Executive Summary

The PRD generator skill incorrectly adds Quality Gates (lint, typecheck, unit tests) to stories that produce no code — spec-only and documentation-only stories. Two root causes compound: (1) the format-rationale keyword detection is fragile, failing on mixed signals like "no executable code" and on rationales that omit doc-only keywords entirely; (2) the PRD metadata section drops `[code-only]` and `[ui-only]` tags from gate names, causing downstream ADO/task consumers to treat every gate as unconditional. This initiative fixes both the classification logic in SKILL.md and the tag preservation in metadata persistence, so that doc-only stories never receive false-positive quality gates and downstream consumers can apply their own tag-based filtering.

### 2. Business Problem & Opportunity

#### Current State

When the PRD skill generates a document where all stories are spec/documentation changes, every story still receives a `**Quality Gates:**` section with lint, typecheck, and unit test ACs. This produces false-positive acceptance criteria that:

**Qualitative Evidence:**
- Engineers see "Lint passes" and "Unit tests pass" ACs on stories that change a markdown spec file — confusing and misleading
- ADO sync creates work items with quality gate ACs that are impossible to verify against the actual deliverable
- Task skill inflates hour estimates by adding 0.25h per false gate per story
- The 004 BRD_PRD (Feature Description Template) — a pure spec change — received Quality Gates on all 4 stories

#### Root Cause Analysis

Two independent bugs in SKILL.md Phase 5:

**Bug 1 — Fragile keyword detection (SKILL.md lines 247-250):**
The `[code-only]` filter requires the format rationale to contain a doc-only keyword (`documentation`, `spec`, `wiki`, `markdown`) AND contain none of the code signals (`code`, `implementation`, `UI`, `interface`, `browser`). This fails in three ways:
- Rationales like "no executable **code** is produced" contain the word `code` → mixed-signal → treated as code story (false positive)
- Rationales like "graceful degradation behavior" contain no doc-only keywords at all → gates included by default (false positive)
- The skill both generates and parses the rationale, but has no feedback loop ensuring the generated text uses the expected keywords

**Bug 2 — Tag stripping in metadata (SKILL.md lines 256-264):**
The Document Metadata `Quality Gates:` list drops `[code-only]` and `[ui-only]` annotations. The quality-gates.md spec says tags must be preserved (lines 56-58 show tags in the example), but the generated output omits them. Without tags, downstream consumers (ADO skill, task skill) cannot apply their own filtering.

### 11. Technical Considerations

- **Scope:** Files changed: `plugins/jx-pm/skills/prd/SKILL.md` (classification + metadata rules), `plugins/jx-pm/skills/prd/references/unified-template.md`, `plugins/jx-pm/skills/prd/references/lite-template.md`, and `plugins/jx-pm/skills/prd/references/prd-template.md` (tag examples in templates)
- **Classification approach options:**
  - Option A: Require an explicit `[doc-only]` annotation in the format rationale (structured signal)
  - Option B: Improve keyword detection to handle negation ("no code", "not code-producing") and require at least one doc-only keyword
  - Option C: Add a story-type field to the user story format (e.g., `*Type: doc-only*`) separate from the format rationale
- **Tag preservation:** Update the metadata persistence instruction to include tags verbatim from the resolved profile
