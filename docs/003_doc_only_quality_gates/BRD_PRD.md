## Part I: Strategic Foundation (BRD)

### 1. Executive Summary

The PRD generator (`/jx-pm:prd`) appends three code quality gates â€” Lint passes, Typecheck passes, Unit tests pass â€” to every user story regardless of story type. For documentation-only stories (spec updates, wiki edits, markdown config changes), these gates add no value and inflate AC counts by 3 per story.

This initiative introduces a `[code-only]` tag in the shared quality-gates configuration and story-type detection logic in the PRD skill. Code quality gates will be applied only to stories that produce runnable code artifacts. The change is backward-compatible: stories without a format rationale line continue to receive all gates.

Observed impact in FEAT-002: 4 documentation-only stories Ă— 3 quality gate ACs = 12 wasted ACs, all flagged as meaningless by the team.

### 2. Business Problem & Opportunity

#### Current State

The PRD generator's Phase 5 (Quality Gates) appends all non-`[ui-only]` gates to every story unconditionally. The three default gates â€” Lint passes, Typecheck passes, Unit tests pass â€” are code-tooling checks that are meaningless for documentation artifacts.

**Quantitative Evidence:**
- FEAT-002 (ADO Sync AC Field Fix): 4 stories Ă— 3 quality gate ACs = 12 wasted ACs (all 4 stories were spec/doc changes to `ado.md`)
- Every documentation-only PRD story receives 3 valueless quality gate ACs by default

**Qualitative Evidence:**
- Developers skip quality gate ACs for doc stories, creating unchecked "passed" states in ADO
- PMs observing inflated AC counts lose trust in AC completeness metrics

#### Root Cause Analysis

`quality-gates.md` lacks a `[code-only]` tag â€” the counterpart to the existing `[ui-only]` tag. The PRD skill has no story-type detection; it applies all non-`[ui-only]` gates uniformly. The format rationale line (already generated per story) provides a reliable, machine-written signal for story-type detection that is not yet used.

### US-003-02: Implement [code-only] Gate Filtering in PRD Skill

**As a** PM using `/jx-pm:prd`
**I want** the PRD generator to skip `[code-only]` gates for stories whose format rationale identifies the deliverable as a documentation artifact
**So that** documentation-only user stories in generated PRDs contain zero meaningless code quality gate ACs

*Format: System State â€” the deliverable is a behavior change in the PRD generator's Phase 5 gate selection logic (update to `SKILL.md`); no executable code is produced*

**Acceptance Criteria:**

**System Behavior:**
- AC-003-05: When a story's format rationale line contains any of the keywords `documentation`, `spec`, `wiki`, or `markdown`, the generator omits all `[code-only]` gates from that story's `**Quality Gates:**` section
- AC-003-06: When a story's format rationale line is absent, all gates are included regardless of tag (backward-compatible default â€” current behavior preserved)
- AC-003-07: When a story's format rationale does not match documentation keywords, `[code-only]` gates are included as before
- AC-003-08: When a story is identified as a UI story (format rationale mentions "UI", "interface", or "browser"), both `[code-only]` gates and `[ui-only]` gates are included
- AC-003-09: When ADO or task tools consume an existing PRD with `Quality Gates:` metadata already persisted, they use that metadata verbatim â€” the new filtering logic does not retroactively alter consumed output

**Validates:** OBJ-003-01, OBJ-003-02

### 11. Technical Considerations

- **Detection signal**: The format rationale line (`*Format: Rule-Based â€” the deliverable is a documentation artifact*`) is written by the PRD generator itself â€” reliable and consistent, not user-freeform text
- **Filtering location**: Phase 5 of `plugins/jx-pm/skills/prd/SKILL.md` â€” keyword check before emitting each gate with a `[code-only]` tag

- **Keyword extension**: The doc-detection keyword set can be expanded in future sprints without changing the tag mechanism or downstream tools
